Assignment 1: Shader Live Coding

Code

For this assignment I wanted to try and create a sort of relaxing visual accompaniment to one of my favorite piano pieces, "omake pfadlib" by Hiroyuki Sawano. Whenever I listen to this piece I visualize cooler colors and softer movement at the beginning, with more intense and frenzied movement coming in as the song becomes more emotional. I also experimented with transitioning into more warmer, passionate tones as the music progresses. The entire song is about 3 minutes long, so unfortunately this isn't a fully choreographed live-coding session, but I hope I was able to achieve a little bit of the mesmerizing/relaxing atmosphere I was going for. Technically, I wanted to focus on creating fluid, more natural-movement, with some randomization in the color and movement either being based on time or on the audio.


The feedback I received about this coding session matched more or less with what I was going for - my partner described it as having a "calm aesthetic", with movement and ripples that resembled water - which I wasn't actually going for, but it works in my favor. I also thought it was interesting how he saw the transition from cool to warm colors as one from water to fire, as opposed to just a reflection of the transition from slower to more intense music.

Assignment 2: Analog to Digital Shader Implementation

  • Code
  • Analog Tool

I was really inspired by the mesmerizing movement of squares and diamond shapes across the screen that occurred when playing around with the Vidiot, so my aim for this assignment was to try to replicate those kinds of shapes, along with pleasing color palette choices. My favorite thing about the Vidiot was how tactile it was though - there is something very satisfying about turning physical knobs and pressing buttons and creating a new and unique animation at each move. Therefore, I tried to make as many options for changing the animation available to the user as I could in limited time. If there is something I wish I had really added, it would've been the option to change the animation shapes entirely from squares and diamonds to lines and clearer waves.


The feedback I received about this animation commented mostly on how much the animations focused on the crisscrossing of lines and squares, as well as how the combonation of colors and lines complement each other. One viewer commented that it was a little difficult to discern what some of the controls' Vidiot equivalents were, which is unfortunate but not unexpected, as I found it difficult to draw exact comparisons between specific controls myself. Another viewer commented that he really enjoyed the aesthetic and the "stacking" effect of the grids and boxes created, which I was very happy to hear, and suggested including some blur around the edges of the animation and more definition of the vertical lines to improve the overall look. If I were to go in and revise this project, I'd focus more on creating clearer and more distinct animations, as well as isolating and manipulating variables that have a more obvious effect on the animation so as to make the experience of interacting with the interface more satisfying for viewers and whoever wanted to use this tool.

Assignment 4: Simulation Programming

  • Code
  • Ant Tool

For this assignment I chose to recreate Langton's Ants in glsl, and I was aiming just to get the full simulation working to the best of my ability. Eventually I came up with 2 implementations, though neither works entirely correctly. The first implementation is the one currently shown, which uses the shader sim.glsl . This implementation has each cell check, on each tick, whether it is currently an ant or whether an ant is about to go into itself, and if so, the cell then assigns itself an ant with a random direction value. The second implementation, which uses the shader sim_2.glsl , aims for a more correct algorithm where the cell chooses the direction of its new ant based on its own red value (1.0 for white, 0.0 for black). Both shaders stored the information for each cell as follows: red value signified if the cell was "dead" or "alive", green value signified if the cell had an ant in it or not, and blue signified the direction of the ant. Though the second shader has an algorithm closer to what the actual rules for langton's ants are, I couldn't quite get the simulation to run fully - it just jitters the ants back and forth. The first shader, while a more simpler implementation, does still achieve an ant-like movement across the screen, if a little too uniform in its movements.


The feedback I received in regards to this tool was that, correct algorithm or not, the tool certainly looks like it's producing a bunch of ants, and the way they moved made the viewer quite uncomfortable - an unfortunate reaction to the news that the simulation looked like it should. However, my partner also mentioned that it is sometimes more obvious to see that the ants can move very similarly, and clicking to add ants doesn't always add the ants in the correct position.

Final: Integrating Leap Motion with Game of Life

I had issues uploading this code to Github, so I'm going to explain the relative process of how this works below.

The leap motion hand setup I adapted from a project I completed in a previous class. To view that project, go here. The interaction between hands and the shader I achieved by attaching the Leap Motion ExtendedFingerGesture script to each hand's index finger, which would then trigger the AddNewLifeHand script every time either of the index fingers were "poking" the shader. This AddNewLifeHand script is an adapted version of the AddNewLife script provided, and includes a function which randomly places new life on the shader on every finger poke.

My goal for this project was to get some sort of interaction working between the Leap Motion Controller and the game of life shader present in the example project provided in class. I focused mostly on just getting the motion controllers working, and from there looked into how I could have the user's hands affect the simulation. With the time I had available, I was able to have the user add new life to the game of life by "poking" at the shader in real time, which turns out to be a pretty fun feature. If I had more time to become more familiar with Unity's shader language I would have liked to sync things such as the simulation color and speed to the user's hand motions as well - unfortunately this was my first time ever looking at Unity Cg, and I found it difficult to parse.


The feedback I received for this project was that the person found it fun to poke at the simulation and see changes occur, but they wished for more ways to interact with the shader. In general, they said they spent about half the time just playing around with their hands and seeing how sensitive the controller was. They commented, however, that the background and general aesthetics of the project were nice.
